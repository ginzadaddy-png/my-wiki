게임을 실제 플레이어에게 보여주고 피드백을 수집하는 방법론. 반짝임을 검증하고 치명적 실수를 사전에 잡아내는 핵심 도구.

스튜디오별 피드백 수집 시점·대상·규모를 나란히 비교한 표는 → [[player-feedback|플레이어 피드백 수집 전략 비교]] 참고 (팀 아소비·서커펀치·라리안·민트로켓).

**올바른 테스트 방법 (크리스 짐머만)**
- 무엇을 테스트하는지, 어떻게 측정할지 사전에 명확히 정하기
- 기초 설명 후 **입 다물기**
- 플레이어가 헤맬 때 개입하지 않기 — "그 순간 플레이어는 게임의 실제 모습을 알려주고 있다"
- 잘못된 방법: 어깨 너머로 가르쳐주고 "어떠셨어요?" 묻기

**서커펀치의 두 가지 테스트**
- **내부 테스트**: 전 팀원이 현재 빌드를 플레이 + 구글 시트 피드백. 6개월마다.
  - 효과 ① 모멘텀: 기능을 혼자 붙들지 않고 팀 전체 앞에 내놓아야 한다는 긴장감
  - 효과 ② 팀 전체 이해: 원격 환경에서 게임 전체를 경험하게 하는 가장 강력한 방법
  - 요테이: 28회, 100개 이상 항목, 8인년 플레이 데이터
- **외부 테스트**: 소니 UX 연구소, 타깃 관객과 유사한 테스터. 표정 기록, 라운드테이블 토론
  - 요테이: 19회 대규모 외부 테스트. 가장 긴 단일 테스트 7일 연속
  - 발견 사례: 성소 미발견 → 강제 유도 대신 배치·힌트 개선 (플레이어 주체성 보존)

**팀 아소비의 플레이테스팅**
- 6주마다 12명 일반인 외부 플레이테스트
- 버튼 혼동 구간, 시선 분산 구간 잡아내는 데 집중

**엠바크 스튜디오의 UXR 기반 테스트 (아크 레이더스)**
- 내부 확신 vs 외부 반응의 극단적 불일치를 UXR로 확인: 개발팀 "매우 깊이 있다" / 외부 플레이어 반대
- 데이터가 피드백의 실체를 파악: "PVP가 싫다" → 실제로는 무기 밸런스·매치메이킹 문제
- 숫자가 논쟁을 대체: 루팅 만족도 1.4/3점 → 원인 파악 → 구체적 개선
- 플레이테스트 요청에도 의도 필요: "더 어둡게 해달라" → "의도와 데이터가 있나요?"
- → [[data-driven-development|데이터 기반 개발]] 참고

> 💡 **핵심 인사이트:** "우리가 특별하다고 생각한 것들이 실제로 특별한가"를 확인하는 유일한 방법은 내가 아닌 사람이 플레이하는 것을 보는 것이다. 그리고 그 결과를 두려워하지 않고 인정하는 것이 가장 어렵다 — 엠바크는 3년이 걸렸다.

## 규모 선택 프레임워크

플레이테스트는 **하나의 방법이 아니라 7단계 스펙트럼**이다. 단계마다 답할 수 있는 질문이 다르고, 비용·시간·노출 위험도 다르다. 잘못된 단계를 선택하면 답을 못 얻을 뿐 아니라 자원·평판을 낭비한다.

**7단계 스펙트럼**

| 단계 | 규모·주기 | 답할 수 있는 질문 | 대표 사례 |
|---|---|---|---|
| 1. 셀프 도그푸딩 | 개발자 1명, 매일 | "내가 보기에 재미있나", 빌드 깨짐 | [[hazelight-studios]] 요세프 파레스 — 외부 노출 전 공동 디렉터와 게임 전체를 직접 코옵 플레이 |
| 2. 소규모 내부 테스트 | 5~15명, 주 단위 | 모먼트 투 모먼트 감각, 컨트롤, 즉각적 혼동 | [[team-asobi]] 화요일 플레이 / [[supergiant-games]] 전팀 플레이스루 |
| 3. 스튜디오 전체 내부 테스트 | 30~200명, 4~8주 | 전체 페이싱, 모멘텀, 통합 버그 | [[sucker-punch-productions]] 28회·8인년 플레이 데이터(요테이) |
| 4. 폐쇄형 외부 랩 테스트 | 12~50명, 유료·NDA | 튜토리얼 명료성, 첫 시간 이탈, 인구통계학적 반응 | [[sucker-punch-productions]] 소니 UXR / [[team-asobi]] 12명 / [[embark-studios]] UXR |
| 5. 친구·가족·디스코드 폐쇄 베타 | 50~5,000명, 반복 | 장시간 리텐션, 코옵 다이내믹, 소규모 서버 부하 | [[supergiant-games]] 하데스 EA + 디스코드 / [[hazelight-studios]] 폐쇄 코옵 |
| 6. 공개 무료 데모 / Steam Next Fest | 1만~50만, 1회성 | 마케팅 시그널, 위시리스트 전환, "이 콘셉트가 차갑게 닿는가" | [[sandfall-interactive]] 클레르 옵스퀴르 33원정대 (2024 Next Fest) |
| 7. 유료 얼리 액세스 / 공개 알파·베타 | 10만~500만, 수개월~수년 | 시스템 밸런스, 콘텐츠 수명, 모네타이제이션, 서버 스케일링 | [[larian-studios]] [[baldurs-gate-3]] (3년 EA, Act 1만) / [[iron-gate-studio]] 발하임 / Bungie 데스티니 스트레스 베타 |

**단계 × 질문 매트릭스**

| 프로젝트 단계 | 핵심 질문 | 권장 단계 |
|---|---|---|
| 프리프로덕션·프로토타입 | 코어 동사가 재미있나 | 1~2 |
| 첫 플레이어블 | 처음 보는 사람이 10분 안에 이해하나 | 2~4 |
| 버티컬 슬라이스 | 첫 접촉의 "와우"가 작동하나 | 3~4 |
| 알파 (콘텐츠 완료) | 전체 페이싱·튜토리얼 퍼널 | 3~4, 가끔 5 |
| 베타 (폴리시) | 엣지 케이스 버그·퍼포먼스·밸런스 | 4~5, 마케팅 준비됐으면 6 |
| 출시 직전 | 마케팅 시그널 + 최종 밸런스 | 6 (Next Fest) / 7 (라이브 서비스) |
| 출시 후 라이브 서비스 | 지속 밸런스, 콘텐츠 케이던스 | 5~7 상시 |

**선택 수정 요인**
- **리스크 프로파일**: $200M AAA는 후반까지 단계 4(통제된 UXR) 의존, 인디는 단계 3을 건너뛰고 4+6 조합
- **온라인 vs 오프라인**: [[live-service-design|라이브 서비스]]는 단계 7 스트레스 테스트 필수 (Bungie 데스티니 베타는 디자인이 아니라 서버 부하 검증이 1차 목적)
- **장르 관습**: [[soulslike|소울라이크]]는 작은 전문가 그룹에서 난이도 곡선 검증, 라이프 시뮬·내러티브는 단계 5+ 장시간 세션 필요

## 단계 사례 심층

**[[larian-studios|라리안]] [[baldurs-gate-3|발더스 게이트3]] — 3년 EA를 Act 1에만**
액트 1만 EA로 공개해 파티 대화 밀도, 주사위 굴림 감각, 저레벨 클래스 밸런스를 검증. 텔레메트리 기반으로 액트 1을 여러 번 재편집. 액트 2~3는 스토리 서프라이즈 보존을 위해 내부·폐쇄 테스트만 거침. **EA는 마케팅 도구가 아니라 검증 도구** — 스벤 빈케 공개 발언.

**[[iron-gate-studio|아이언 게이트]] 발하임 — EA 우선 전략**
5인 팀이 대규모 멀티플레이어 서버 부하를 내부적으로 시뮬레이션 불가 → 공개 EA가 퍼블리셔 QA 조직 역할 대체. 헨리크 톤퀴스트 인터뷰.

**[[supergiant-games|슈퍼자이언트]] 하데스 — 디스코드 주도 EA**
주간 내부 빌드 → 월간 EA 패치 → 포럼·디스코드 트리아지. **"콘텐츠 더 달라" 요구는 의도적으로 무시하고 코어 루프 반복성에만 집중**. 그렉 카사빈 GDC 2021 강연.

**[[hazelight-studios|하즐라이트]] 잇 테이크스 투·스플릿 픽션**
요세프 파레스가 외부 테스트 전 공동 디렉터와 게임 전체를 여러 번 코옵 플레이. 이후 5쌍 정도의 낯선 짝 세션. **타이트한 내러티브 코옵에 대규모 공개 베타에 회의적** — 스포일러만 나고 페이싱 문제는 5쌍이나 5,000쌍이나 비슷하게 드러남.

**Steam Next Fest의 정체**
크리스 즈코프스키(How to Market a Game) 데이터: 데모 CTR-위시리스트 전환이 핵심 지표. **Next Fest는 마케팅 이벤트지 디자인 피드백 이벤트가 아니다.** 권장 80% 폴리시드 / 20% 게임 본편 대표 콘텐츠.

## 안티패턴

1. **조기 공개 테스트** — 단계 4가 첫 10분을 정리하기 전에 데모 출시. 위시리스트 전환은 되돌릴 수 없음. 인디 흔한 실패
2. **내부 에코 챔버** — 개발자는 너무 관대함, 깨진 UI에 의도를 읽음. [[embark-studios|엠바크]] 아크 레이더스가 최근 대표 사례 — 내부 "매우 깊다" / 외부 강하게 반대
3. **검증 아닌 마케팅으로서의 EA** — EA를 단순 매출 채널로 취급, 텔레메트리 무시. 실패한 EA 다수의 공통 패턴
4. **친구·가족 편향** — 폐쇄 디스코드 테스터는 장르 팬 자가선택, 기본기를 용서함. 단계 4 콜드 스트레인저 라운드와 반드시 병행
5. **잘못된 질문 규모** — "튜토리얼이 명료한가"를 답하려 5,000명 베타 운영. 12명 랩에서 하루면 답 나옴. **규모는 질문 구체성과 매칭**

## 인용된 프레임워크

- **크리스 짐머만 ([[sucker-punch-productions|서커펀치]]) — "두 가지 테스트 패턴"** — 내부(모멘텀 + 팀 전체 이해) + 외부(타깃 관객 반응). 가장 많이 인용되는 현대 프레임워크
- **"입 다물고 보기"** — 짐머만, 팀 아소비도 동일. 브리핑 후 침묵, 그들이 헤매는 순간이 데이터
- **[[embark-studios|엠바크]] "의도 + 데이터" 원칙** — 가설과 측정 계획 없는 피드백 요청 금지. "더 어둡게"식 입력 차단
- **즈코프스키 80/20 데모 규칙** — 80% 폴리시드 / 20% 본편 대표. 데모는 마케팅용, 디자인 검증용 아님

> 💡 **핵심 원칙:** 규모는 질문 구체성과 매칭한다. "코어 루프가 재미있나"는 단계 1~2, "튜토리얼이 명료한가"는 단계 4, "서버가 100만 동접을 견디나"는 단계 7. 잘못된 단계를 선택하면 답을 못 얻을 뿐 아니라 평판·자원을 낭비한다.
