게임을 실제 플레이어에게 보여주고 피드백을 수집하는 방법론. 반짝임을 검증하고 치명적 실수를 사전에 잡아내는 핵심 도구.

**올바른 테스트 방법 (크리스 짐머만)**
- 무엇을 테스트하는지, 어떻게 측정할지 사전에 명확히 정하기
- 기초 설명 후 **입 다물기**
- 플레이어가 헤맬 때 개입하지 않기 — "그 순간 플레이어는 게임의 실제 모습을 알려주고 있다"
- 잘못된 방법: 어깨 너머로 가르쳐주고 "어떠셨어요?" 묻기

**서커펀치의 두 가지 테스트**
- **내부 테스트**: 전 팀원이 현재 빌드를 플레이 + 구글 시트 피드백. 6개월마다.
  - 효과 ① 모멘텀: 기능을 혼자 붙들지 않고 팀 전체 앞에 내놓아야 한다는 긴장감
  - 효과 ② 팀 전체 이해: 원격 환경에서 게임 전체를 경험하게 하는 가장 강력한 방법
  - 요테이: 28회, 100개 이상 항목, 8인년 플레이 데이터
- **외부 테스트**: 소니 UX 연구소, 타깃 관객과 유사한 테스터. 표정 기록, 라운드테이블 토론
  - 요테이: 19회 대규모 외부 테스트. 가장 긴 단일 테스트 7일 연속
  - 발견 사례: 성소 미발견 → 강제 유도 대신 배치·힌트 개선 (플레이어 주체성 보존)

**팀 아소비의 플레이테스팅**
- 6주마다 12명 일반인 외부 플레이테스트
- 버튼 혼동 구간, 시선 분산 구간 잡아내는 데 집중

**엠바크 스튜디오의 UXR 기반 테스트 (아크 레이더스)**
- 내부 확신 vs 외부 반응의 극단적 불일치를 UXR로 확인: 개발팀 "매우 깊이 있다" / 외부 플레이어 반대
- 데이터가 피드백의 실체를 파악: "PVP가 싫다" → 실제로는 무기 밸런스·매치메이킹 문제
- 숫자가 논쟁을 대체: 루팅 만족도 1.4/3점 → 원인 파악 → 구체적 개선
- 플레이테스트 요청에도 의도 필요: "더 어둡게 해달라" → "의도와 데이터가 있나요?"
- → [[data-driven-development|데이터 기반 개발]] 참고

> **핵심 인사이트:** "우리가 특별하다고 생각한 것들이 실제로 특별한가"를 확인하는 유일한 방법은 내가 아닌 사람이 플레이하는 것을 보는 것이다. 그리고 그 결과를 두려워하지 않고 인정하는 것이 가장 어렵다 — 엠바크는 3년이 걸렸다.
